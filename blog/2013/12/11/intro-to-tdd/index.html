Intro to TDD

Dec 11th, 2013

The past few weeks I’ve been working on programming an a more object oriented way, and I’ve found it to be extremely helpful in solving larger, more difficult problems. This past week I began Test Driven Development using unit tests with Rspec, and after the initial difficulties of learning how to write appropriate tests, I love it. Using objects and unit tests allows you to lower your cognitive load on any given problem by decreasing the number of variables and outcomes you need to store in your working memory at any given time. Keeping track of less lets you do more, and with well laid out objects and tests your code is more readable for your pair partner.

Testing has obvious benefits for code maintainability. What’s not obvious is it’s effect on productivity. On the surface, writing tests upfront seems to double the work. And perhaps for *very* small projects this is true. In the long run, however, I am convinced it helps increase productivity. When I have to read a codebase I had written earlier with the intent of refactoring/expanding the functionality, I give up some of my working memory to understanding what’s going on. With my remaining cognitive capacity I build what I planned to build. As the codebase becomes larger or has more tough spots, my ability to make improvements for a given unit of time decreases. Eventually my ability to make marginal improvements will diminish as my brain struggles to keep track of what the existing codebase is doing if it’s not well encapsulated. The same is true of a team of developers on an ever larger projects. Without a good object oriented approach and unit tests, adding developers will not provide a linear improvement in velocity. With tests and OOP best practices, however, this is not the case.

I look forward to integrating TDD into my regular workflow, and to learning more about OOP practices. I’ve started _Practical Object Oriented Design in Ruby_ by Sandi Metz, and I love how well the principles are explained so far.
